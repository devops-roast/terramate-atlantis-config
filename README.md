Automatically generate `atlantis.yaml` from your Terramate stacks.

## Overview

`terramate-atlantis-config` is a CLI tool that discovers [Terramate](https://terramate.io) stacks in your repository and generates an `atlantis.yaml` configuration file for [Atlantis](https://www.runatlantis.io). It's built for teams running Terramate alongside Atlantis who want to avoid hand-maintaining project lists as their stack count grows.

Instead of updating `atlantis.yaml` by hand every time you add, rename, or remove a stack, you run this tool in CI and let it do the bookkeeping.


## Quick Start

```sh
terramate-atlantis-config generate --root . --output atlantis.yaml
```

Sample output:

```yaml
# Generated by terramate-atlantis-config. DO NOT EDIT.
version: 3
automerge: false
parallel_plan: true
parallel_apply: true
projects:
    - name: aws-production-vpc
      dir: stacks/aws/production/vpc
      autoplan:
        enabled: true
        when_modified:
            - '*.tf'
            - '*.tf.json'
            - .terraform.lock.hcl
```

## Configuration

### CLI Flags

| Flag | Default | Description |
|------|---------|-------------|
| `--root` | `.` | Root directory to search for stacks |
| `--output` | stdout | Output file path |
| `--autoplan` | `true` | Enable autoplan for all projects |
| `--parallel` | `true` | Enable parallel plan and apply |
| `--automerge` | `false` | Enable automerge in atlantis.yaml |
| `--workflow` | | Default workflow name for all projects |
| `--terraform-version` | | Default Terraform version for all projects |
| `--filter` | | Include only stacks matching this tag filter |
| `--filter-path` | | Include only stacks under this path |
| `--exclude-tags` | | Exclude stacks with these tags |
| `--exclude-path` | | Exclude stacks under this path |
| `--changed` | `false` | Only include stacks changed vs base ref |
| `--changed-base-ref` | `main` | Git ref to compare against for change detection |
| `--create-project-name` | `true` | Generate a project name from the stack path |
| `--project-name-strategy` | `auto-strip` | Naming strategy: `auto-strip`, `stack-name`, `last-n`, `full` |
| `--project-name-prefix` | | Strip this prefix from stack paths when generating names |
| `--project-name-depth` | `3` | Number of trailing path segments for `last-n` strategy |
| `--create-workspace` | `false` | Set workspace from stack path |
| `--execution-order-groups` | `false` | Add execution order groups from stack dependencies |
| `--depends-on` | `false` | Add depends_on from stack dependencies |
| `--when-modified` | `*.tf, *.tf.json, .terraform.lock.hcl` | File patterns that trigger autoplan |
| `--apply-requirements` | | Apply requirements for all projects |
| `--tag-workflow` | | Map stack tags to workflows |
| `--tag-requirements` | | Map stack tags to apply requirements |
| `--preserve-workflows` | `false` | Keep existing workflow blocks from current atlantis.yaml |
| `--generate-workflows` | `false` | Auto-generate workflow blocks wrapping terraform in `terramate run` |
| `--workflow-terramate-wrap` | `true` | Wrap terraform commands with `terramate run` in generated workflows |
| `--sort-by` | `dir` | Sort projects by this field |
| `--format` | `yaml` | Output format (`yaml` or `json`) |
| `--no-header` | `false` | Omit the generated-by header comment |
| `--diff` | `false` | Print a diff of the current vs generated output |
| `--check` | `false` | Exit non-zero if the output would change (for CI) |
| `--config` | | Path to config file |

### Config File

Place a `.terramate-atlantis.yaml` file in your project root. All CLI flags can be set here. CLI flags take precedence over config file values.

```yaml
output: atlantis.yaml
workflow: terramate
autoplan: true
parallel: true
project_name_strategy: last-n
project_name_depth: 2
apply_requirements:
  - approved
  - mergeable
when_modified:
  - "*.tf"
  - "*.tf.json"
  - ".terraform.lock.hcl"
tag_workflows:
  production: prod-workflow
  staging: staging-workflow
workflows:
  prod-workflow:
    plan:
      steps:
        - run: terramate run -- terraform init -input=false
        - run: terramate run -- terraform plan -input=false -out=$PLANFILE
    apply:
      steps:
        - run: terramate run -- terraform apply $PLANFILE
```

## Per-Stack Customization via Globals

Set Terramate globals in any stack to override the defaults for that project.

| Global | Type | Description |
|--------|------|-------------|
| `atlantis_skip` | bool | Skip this stack entirely |
| `atlantis_workflow` | string | Override workflow for this stack |
| `atlantis_autoplan` | bool | Override autoplan enabled/disabled |
| `atlantis_terraform_version` | string | Override Terraform version |
| `atlantis_when_modified` | list(string) | Override when_modified patterns |
| `atlantis_extra_deps` | list(string) | Additional when_modified entries merged with defaults |
| `atlantis_apply_requirements` | list(string) | Override apply requirements |

Example `stack.tm.hcl`:

```hcl
globals {
  atlantis_workflow          = "production"
  atlantis_terraform_version = "1.5.0"
  atlantis_extra_deps        = ["../modules/**/*.tf"]
}
```

## Features

### Project Naming Strategies

By default, project names are derived from the full stack directory path, which can produce long names in deep hierarchies. Four naming strategies are available:

| Strategy | Flag | Behavior | Example (`stacks/aws/prod/eu-west-1/db/billing`) |
|----------|------|----------|--------------------------------------------------|
| `auto-strip` | default | Strips longest common path prefix across all stacks | `prod-eu-west-1-db-billing` (if all stacks share `stacks/aws`) |
| `stack-name` | `--project-name-strategy stack-name` | Uses the stack's `name` from `stack {}` block with parent context | `eu-west-1-db-billing` |
| `last-n` | `--project-name-strategy last-n` | Uses last N path segments (default: 3) | `db-billing` (depth=2) |
| `full` | `--project-name-strategy full` | Full path, original behavior | `stacks-aws-prod-eu-west-1-db-billing` |

All strategies include **automatic disambiguation**: if two stacks would get the same name, additional parent segments are added until names are unique.

You can also strip an explicit prefix with `--project-name-prefix`:

```sh
terramate-atlantis-config generate --project-name-prefix "stacks/aws" ...
```

**Recommended config for `--changed` workflows** (small project sets per PR, collision is rare):

```yaml
# .terramate-atlantis.yaml
project_name_strategy: last-n
project_name_depth: 2
```

### Change Detection

Pass `--changed --changed-base-ref main` to only generate projects for stacks that have changed relative to a git ref. Useful in PR workflows where you want Atlantis to plan only what's affected.

```sh
terramate-atlantis-config generate --root . --output atlantis.yaml \
  --changed --changed-base-ref main
```

### Dependency Tracking

`--execution-order-groups` adds execution order group numbers derived from Terramate's stack dependency graph. `--depends-on` populates the `depends_on` field for each project. Both flags read the dependency information Terramate already tracks.

### Workflow Generation

`--generate-workflows` creates workflow blocks in the output that wrap Terraform commands with `terramate run`. Combine with `--workflow-terramate-wrap=false` if you want the workflow structure without the wrapper.

### CI Integration

**Check mode** exits non-zero if the generated output differs from what's on disk. Use this to enforce that `atlantis.yaml` is always up to date:

```sh
terramate-atlantis-config generate --root . --output atlantis.yaml --check
```

**Diff mode** prints a unified diff of the current file vs the generated output, useful for PR review:

```sh
terramate-atlantis-config generate --root . --output atlantis.yaml --diff
```

## GitHub Action

```yaml
- uses: devops-roast/terramate-atlantis-config@v1
  with:
    root: "."
    output: atlantis.yaml
    check: "true"
```

Full CI example that fails the build if `atlantis.yaml` is out of sync:

```yaml
jobs:
  check-atlantis-config:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: devops-roast/terramate-atlantis-config@v1
        with:
          root: "."
          output: atlantis.yaml
          check: "true"
```
